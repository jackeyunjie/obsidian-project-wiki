# 数据库选型调研：MySQL 8 vs PostgreSQL 16

**日期**：2026-06-22  
**作者**：技术负责人  
**状态**：已完成

---

## 背景

社区养老平台 MVP 需要一款关系型数据库存储用户、打卡、工单等结构化数据。

## 对比维度

### 1. 性能

| 场景 | MySQL 8 | PostgreSQL 16 |
|------|---------|---------------|
| 简单 CRUD（高并发） | 优秀 | 优秀 |
| 复杂 JOIN 查询 | 良好 | 更优（更智能的查询优化器） |
| JSON 字段操作 | 良好（JSON 类型） | 优秀（JSONB + 索引） |
| 全文搜索 | 一般（需 InnoDB FTS） | 优秀（内置 tsvector） |

### 2. 运维成本

| 维度 | MySQL 8 | PostgreSQL 16 |
|------|---------|---------------|
| 团队熟悉度 | ★★★★★ | ★★☆☆☆ |
| 云服务商支持 | 所有主流云 | 所有主流云 |
| 备份工具 | mysqldump / xtrabackup | pg_dump / WAL |
| 监控生态 | 成熟 | 成熟 |

### 3. 特性对比

| 特性 | MySQL 8 | PostgreSQL 16 |
|------|---------|---------------|
| CTE / 窗口函数 | ✓ | ✓ |
| JSON 支持 | ✓（JSON） | ✓✓（JSONB） |
| 分区表 | ✓ | ✓（更灵活） |
| 扩展性 | 一般 | 优秀（PostGIS、pgvector） |

## 结论

**推荐 MySQL 8**，理由：
1. 团队有 3 年以上 MySQL 运维经验，上手成本低
2. MVP 阶段无复杂查询需求，MySQL 性能完全够用
3. 云服务（阿里云 RDS MySQL）价格合理

**后续考虑**：如果项目进入增长期，需要 pgvector（向量搜索）或 PostGIS（地理信息），可评估迁移至 PostgreSQL。
