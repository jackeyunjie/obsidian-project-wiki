## 议题

### 1. 项目背景与目标

社区养老数字化平台「吾家有老」面向社区服务站、老人及家属三方，MVP 阶段聚焦：
- 老人每日打卡（健康状态上报）
- 家属实时查看老人状态
- 紧急呼叫响应
- 生活服务工单（代买、手机教学等）

### 2. 技术选型讨论

| 组件 | 候选方案 | 最终决定 |
|------|----------|----------|
| 后端框架 | NestJS vs Koa vs Express | **NestJS**（模块化、TypeORM 原生支持） |
| 数据库 | MySQL 8 vs PostgreSQL 16 | **MySQL 8**（运维团队更熟悉） |
| 缓存 | Redis vs Memcached | **Redis**（支持发布订阅、队列） |
| 小程序 | 微信原生 vs Taro vs uni-app | **微信原生**（MVP 阶段最轻量） |
| 部署 | Docker Compose vs K8s | **Docker Compose**（MVP 够用，后续可迁 K8s） |

> 详细调研见 [[adr-001-choose-postgresql]]（注：最终选了 MySQL，但 PostgreSQL 调研仍有参考价值）

### 3. MVP 范围确认

**包含**：
- 老人端小程序（打卡 + 紧急呼叫 + 服务求助）
- 家属端小程序（状态查看 + 异常提醒 + 服务下单）
- 后端 API（认证、打卡、提醒、工单）
- 基础管理后台（门店管理、老人管理）

**不包含**（后续迭代）：
- 微信支付
- 语音外呼
- AI 健康分析
- 多语言支持

### 4. 里程碑

| 阶段 | 时间 | 目标 |
|------|------|------|
| M0 | W1-W2 | 脚手架搭建 + 数据库设计 |
| M1 | W3-W4 | 核心 API + 老人端打卡流程 |
| M2 | W5-W6 | 家属端 + 异常提醒 |
| M3 | W7-W8 | 服务工单 + 联调测试 |

### 5. 风险与约束

- **微信审核**：小程序类目需「社区服务」或「医疗健康」，需提前准备资质
- **数据合规**：老人手机号等敏感信息需加密存储（AES-256-GCM）
- **人力**：目前仅 1 名后端 + 1 名前端，需控制 MVP 范围
